# 心情怪兽收容所｜技术栈与开发路线 v0.1

> Mood Monster Shelter · 微信小程序 / App 技术选型建议

**适用目标：** 一个人快速开发、手机端可用、便于分享传播、能展示产品 / 设计 / Coding / Agent 能力。

> **推荐结论：第一版优先做微信小程序,不建议直接做原生 App。**

**推荐技术栈:Taro + React + TypeScript + CloudBase 云开发。**

**理由:** 心情怪兽收容所是轻量、有梗、依赖分享传播的作品。小程序打开成本低,天然适合朋友圈、微信群和二维码传播;Taro + React 又能体现现代前端工程能力,后续也能扩展 H5 或 App。

---

## 1. 技术路线总览

| 层级 | 推荐方案 | 说明 |
|---|---|---|
| 端形态 | 微信小程序优先 | 更适合手机使用、快速发布和社交分享。 |
| 前端框架 | Taro + React + TypeScript | 组件化强,适合作品集展示,也保留多端扩展能力。 |
| 样式方案 | CSS Modules / Less / NutUI React | 先保证开发效率,后续再做高度定制 UI。 |
| 动画 | CSS Animation + Taro 动画 API | 用于收容动画、卡片入场、怪兽捕获等轻动效。 |
| 后端 | CloudBase 云开发 | 免服务器,适合个人开发者快速上线。 |
| 数据库 | CloudBase Database + 本地 Storage | MVP 可先本地存图鉴,后续再云端同步。 |
| 云函数 | CloudBase Cloud Functions | 负责调用大模型,保护 API Key。 |
| AI 接口 | DeepSeek / Qwen / Kimi / 智谱 GLM / OpenAI-compatible | 优先选中文表达好、成本低、结构化输出稳定的模型。 |
| 分享卡 | 小程序 Canvas 生成图片 | 生成今日怪兽卡、借口死亡证明、怪兽出院证明。 |

---

## 2. 为什么第一版做小程序,而不是直接做 App

这个产品的核心传播链路应该是:**朋友分享一张怪兽卡 → 我点进去玩一下 → 生成自己的怪兽 → 再分享出去**。这个链路里,“下载 App”会成为明显阻力。

小程序更适合轻量尝鲜、社交裂变和作品发布。用户不需要安装,扫码或点击就能进入;你也能更快把 MVP 放出去收反馈。

原生 App 更适合长期产品,不适合作为第一版验证。它会增加应用市场审核、iOS / Android 适配、包体积、下载转化和发布周期等成本。

---

## 3. 推荐方案:Taro + React + TypeScript + CloudBase

这套方案的优势是:开发速度快、工程结构清晰、前端能力展示充分、发布路径顺畅,并且后续可以扩展 H5 或移动 App。

### 3.1 技术栈明细

- **前端框架:** Taro + React + TypeScript
- **UI 样式:** CSS Modules / Less / NutUI React
- **动画:** CSS Animation + Taro 动画 API
- **后端:** 腾讯云 CloudBase 云开发
- **数据库:** CloudBase Database
- **云函数:** CloudBase Cloud Functions
- **图片存储:** CloudBase Storage
- **AI 接口:** 云函数调用大模型 API
- **本地缓存:** Taro Storage / 微信小程序 Storage
- **发布端:** 微信小程序;后续可兼容 H5

### 3.2 为什么不用原生微信小程序

原生小程序当然能做,但作品集的工程表达不如 React 组件化清晰。Taro 能让你用 React 思维组织页面、组件和状态,代码更适合展示。

你的项目里会有 `MonsterCard`、`CaptureLoading`、`ExcusePanel`、`SharePoster`、`MonsterGallery` 等组件,这种结构用 React 写会更舒服,也更像一个现代前端项目。

## 4. 产品技术架构

MVP 的核心架构建议如下:

```
微信小程序前端
  → 输入用户状态
  → 调用 CloudBase 云函数 analyzeMood
  → 云函数携带 API Key 请求大模型
  → 大模型返回结构化 JSON
  → 云函数做安全校验和格式修正
  → 前端渲染怪兽档案 / 借口粉碎 / 5 分钟任务
  → Canvas 生成分享卡
  → 本地 Storage 保存怪兽图鉴
```

> **关键原则:不要在前端直接调用大模型。** API Key 必须放在云函数或后端服务里,前端只调用你自己的云函数接口。

---

## 5. 大模型接入建议

第一版不需要追求最强推理模型。这个产品更看重**中文表达、有梗文案、结构化 JSON 稳定、响应速度和成本**。

可以优先选择 DeepSeek、Qwen、Kimi、智谱 GLM 或任意 OpenAI-compatible API。只要能稳定返回 JSON,并且中文风格可控,就足够支撑 MVP。

后续如果要做更强体验,可以增加**模型路由**:普通请求走低成本模型,高风险情绪输入走更稳健的安全响应链路。

## 6. 怪兽匹配策略:规则表 + LLM 文案生成

第一版不建议把怪兽名字完全交给大模型自由生成。建议你**先固定 16 只怪兽,用规则表控制体系稳定性**,再让模型生成具体解释和行动建议。

| 类型 | 输入特征 | 匹配怪兽 | 策略说明 |
|---|---|---|---|
| 伪忙碌 | 我没时间、忙了一天、没进展 | 伪忙碌仓鼠怪 | 用忙碌感替代进展感。 |
| 完美主义 | 还没准备好、再学一阵、再优化 | 完美主义打磨怪 | 用准备感逃避发布和评价。 |
| 信息囤积 | 先收藏、先存一下、资料太多 | 收藏松鼠怪 | 用囤积信息制造掌控感。 |
| 低落自责 | 我好废、我不行、很委屈 | 下雨小狗怪 | 需要被安慰的疲惫感。 |
| 焦虑冻结 | 很慌但动不了、不知道从哪开始 | 焦虑转圈怪 | 大脑同时踩油门和刹车。 |
| 深夜冲动 | 想换工作、换城市、重启人生 | 重启人生巨兽 | 低电量时做重大判断。 |

> 上表为示例匹配规则,完整 16 只怪兽体系详见产品方案文档第 8 节。

## 7. 云函数返回结构建议

云函数 `analyzeMood` 建议返回前端可直接渲染的结构化 JSON,避免前端再做复杂文本解析。

```json
{
  "monsterName": "完美主义打磨怪",
  "monsterType": "借口类",
  "dangerLevel": "中高",
  "catchphrase": "等我准备好了再开始",
  "trueIdentity": "用准备感逃避被评价",
  "whyItAppears": "你担心第一版不够好,所以一直停在准备阶段。",
  "whatItProtects": "它想保护你不被否定。",
  "excuseCrush": "作品不是憋出来的,是迭代出来的。",
  "doNotFeed": "不要再搜第 29 个优秀案例了。",
  "microAction": "打开文档,只写标题和第一句话。",
  "cardText": {
    "title": "今日心情怪兽",
    "subtitle": "完美主义打磨怪",
    "line": "先做一个丑丑的 v0"
  }
}
```

**字段说明:**

| 字段 | 含义 |
|---|---|
| `monsterName` | 怪兽名称(来自固定规则表) |
| `monsterType` | 怪兽类型:情绪类 / 借口类 / 混合类 |
| `dangerLevel` | 危险等级:低 / 中 / 中高 / 高 |
| `catchphrase` | 怪兽口头禅(用户最常说出口的那句) |
| `trueIdentity` | 伪装鉴定:它真正的身份是什么 |
| `whyItAppears` | 为什么它会出现(背后的真实原因) |
| `whatItProtects` | 它在保护你什么(温柔视角) |
| `excuseCrush` | 借口粉碎:一句话拆穿它 |
| `doNotFeed` | 不要再喂它什么(停止强化行为) |
| `microAction` | 5 分钟驯化任务(可执行的小行动) |
| `cardText` | 分享卡文案:`title` / `subtitle` / `line` |

## 9. 开发顺序

1. **纯前端假数据版:** 先不接大模型,用 mock 数据跑通首页、加载页、结果页、分享页、图鉴页。重点先验证体验。
2. **接 CloudBase 云函数:** 前端调用 `analyzeMood(inputText)`,云函数先返回固定 JSON,确认链路正常。
3. **接入大模型:** 云函数中调用大模型,要求模型返回结构化 JSON,并做基础格式校验。
4. **做分享卡生成:** 用 Canvas 生成今日怪兽卡、借口死亡证明和怪兽出院证明,这是传播核心。
5. **做图鉴保存:** 先用本地 Storage 保存历史怪兽,不做登录,降低开发复杂度。
6. **优化动画和视觉:** 补充收容动画、卡片动效、怪兽头像和危险等级标签。

## 11. 如果后续要做 App

如果小程序验证不错,第二阶段可以考虑 App。但建议**不要直接做原生 iOS / Android**。

推荐 **React Native + Expo + TypeScript**。优点是开发效率高,可以复用 React 思维,也更适合一个人做 iOS / Android 双端。

Flutter 也可以,但需要学习 Dart,前期对你这个作品来说性价比不如 React 系技术栈。

如果只是为了作品集展示,先补一个 **H5 官网**,比直接做 App 更有效。

## 12. 风险与注意事项

- **API Key 安全:** 不要把任何大模型 API Key 写在小程序前端。必须通过云函数调用。
- **模型输出不稳定:** 使用 JSON Schema 校验;失败时用兜底文案或重试一次。
- **安全边界:** 涉及自伤、自杀、强烈绝望等输入时,不要继续搞笑化,要进入安全回应。
- **审核风险:** 小程序涉及情绪表达,但不要包装成心理诊断或医疗工具。
- **成本控制:** 限制单次输入长度、控制输出长度、优先使用低成本模型。
- **体验节奏:** 不要让用户等太久;加载过程用“捕获动画”承接等待。

---

## 13. 最终建议

> - 第一版就按 **Taro + React + TypeScript + CloudBase 微信小程序** 做。
> - 不要一开始追求复杂 App,也不要先做登录、社区、排行榜。
> - 先把“**输入 → 收容动画 → 怪兽档案 → 5 分钟任务 → 分享卡 → 图鉴保存**”这条核心链路打磨顺。
> - 这个项目的成败关键不是模型多强,而是结果是否“**准、有梗、有温度、愿意分享**”。
